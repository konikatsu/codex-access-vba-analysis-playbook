# コンテンツの有効化を確認する

## 目的

Access DBを開いたときに、画面上部へセキュリティ警告が出ることがあります。

この状態で「コンテンツの有効化」をクリックしていないと、VBA、マクロ、ActiveX、フォームイベント、COM操作が期待どおり動かない場合があります。

Access/VBA作業で不可解なエラーが出たら、DB破損と判断する前に、まずコンテンツが有効化されているか確認します。

## 典型的な症状

コンテンツ未有効化のままだと、次のような症状が出ることがあります。

- フォームのイベントが動かない。
- ボタンを押しても処理が動かない。
- `Application.Run` が失敗する。
- VBEコンパイルやフォーム保存が失敗する。
- COMからフォームやVBAを保存できない。
- `ActiveVBProject` やVBE操作が不安定に見える。
- `データベースの Visual Basic for Applications プロジェクトが破損しています。` のように見えるエラーが出る。

最後のエラーは、本当にVBAプロジェクトが破損している場合もあります。  
ただし、コンテンツ未有効化が原因で似た状態に見えることがあるため、先に有効化状態を確認します。

## 今回の教訓

Access作業では、次の判断順を間違えると大きく遠回りします。

NGだった判断:

- 「コンテンツの有効化」未確認のまま、VBAプロジェクト破損と見なす。
- `ActiveVBProject = null` の原因を、信頼設定ではなくDB破損やCOM不安定と見なす。
- `AutomationSecurity = 3` を使った状態で、VBA実行やコンパイル可否を判断する。
- 非表示COMで重いフォームを開き、タイムアウト原因を見えないまま推測する。

先に疑うべきもの:

- Accessの黄色バー、つまりアクティブコンテンツ未有効化。
- トラストセンターの `VBA プロジェクト オブジェクト モデルへのアクセスを信頼する` がOFF。
- `AutomationSecurity` の使い分けミス。
- 非表示COM実行中のダイアログ待ち、ODBC接続待ち、フォームイベント停止。

結論:

```text
Accessのセキュリティ/信頼状態の切り分けが先。
DB破損、VBAプロジェクト破損、COM不安定を疑うのはその後。
```

## GUIで確認する

対象DBをAccessで開きます。

画面上部にセキュリティ警告が出ている場合は、次をクリックします。

```text
コンテンツの有効化
```

その後、VBEを開いてコンパイルできるか確認します。

```text
Alt + F11
-> デバッグ
-> コンパイル
```

コンパイルが通るなら、VBAプロジェクト破損扱いはいったん撤回し、通常の切り分けに戻します。

## 信頼済み場所に置く

繰り返しテストする作業コピーでは、毎回セキュリティ警告に対応するより、
作業フォルダをAccessの信頼済み場所に追加する方が安定します。

推奨:

```text
プロジェクトルート、または作業コピーを置く親フォルダ
例: C:\dev\project-name\
```

設定場所:

```text
Access
-> ファイル
-> オプション
-> トラスト センター
-> トラスト センターの設定
-> 信頼できる場所
-> 新しい場所の追加
```

可能なら「サブフォルダーも信頼する」をONにします。

重要:

- `/cmd SKIP_AUTOEXEC` は起動処理をスキップするためのDB側の仕組みであり、セキュリティ警告を解除しません。
- `Start-Process msaccess.exe "...accdb" /cmd SKIP_AUTOEXEC` で警告が出る場合は、起動抑止ではなく信頼設定の問題として扱います。
- VBA/マクロ/フォームイベントの動作テストでは、信頼済み場所または「コンテンツの有効化」が必要です。
- `AutomationSecurity = 3` はマクロを強制無効化するため、動作テストでは使いません。

## COM作業との関係

COMでVBAを実行する場合は、`AutomationSecurity = 1` を `OpenCurrentDatabase` の前に設定します。

```powershell
$access = New-Object -ComObject Access.Application
$access.Visible = $false
$access.AutomationSecurity = 1
$access.OpenCurrentDatabase($dbPath)
$access.RunCommand(126)
```

`AutomationSecurity = 3` は、マクロを止めてGUI確認したい場合の候補です。  
VBA実行、`Application.Run`、コンパイル確認には向かない場合があります。

## VBAプロジェクト オブジェクトモデルへのアクセス信頼

`ActiveVBProject` が `null` になる場合は、次の設定がOFFの可能性があります。

```text
ファイル
-> オプション
-> トラスト センター
-> トラスト センターの設定
-> マクロの設定
-> VBA プロジェクト オブジェクト モデルへのアクセスを信頼する
```

この設定をONにした後は、Accessを全て閉じて開き直します。

注意:

- これは「すべてのマクロを有効にする」とは別の設定です。
- 組織ポリシーで変更できない場合があります。
- 変更できない場合、VBE自動編集は諦めて手動貼り付けに切り替えます。

## 切り分け順

不可解なVBA/COMエラーが出たら、次の順に確認します。

```text
1. 原本ではなく作業コピーを用意する
2. 作業コピーをGUIで開く
3. セキュリティ警告があれば「コンテンツの有効化」をクリックする
4. VBEで手動コンパイルする
5. VBAプロジェクト オブジェクトモデルへのアクセス信頼を確認する
6. 通るなら、破損ではなくセキュリティ状態の問題だった可能性を考える
7. その後、COMでは AutomationSecurity=1 で再確認する
8. それでも失敗する場合だけ、VBAプロジェクト破損やフォーム定義破損を疑う
```

## 注意

- 本体DBで試さず、作業コピーで確認します。
- 失敗したコピーを修復しながら続けず、新しい作業コピーで再確認します。
- セキュリティ警告を無視したまま、`LoadFromText` やフォーム保存の失敗をDB破損と断定しないでください。
- 信頼済みの場所に作業フォルダを置くと、セキュリティ警告を減らせる場合があります。
- `/cmd SKIP_AUTOEXEC` と信頼済み場所は別の対策です。両方必要になる場合があります。
