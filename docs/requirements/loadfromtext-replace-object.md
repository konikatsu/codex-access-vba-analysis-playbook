# フォーム/レポート/モジュールをLoadFromTextで差し替える

## 結論

作業コピー上であれば、フォームやレポートの差し替えは次の手順で進めてよいです。

```text
DeleteObject
-> LoadFromText
-> RunCommand(126)
-> Accessを閉じる
-> 再オープンして確認
```

## 安全手順

```text
本体DBまたは直近の成功コピー
-> 新しい作業コピーを作る
-> 作業コピー上で対象オブジェクトをDeleteObject
-> LoadFromTextで取り込む
-> RunCommand(126)でコンパイル
-> フォーム/レポートを開いて確認
-> 成功したコピーを次の土台にする
-> 失敗したコピーは破棄する
```

## SaveAsText退避は必須か

作業コピーを毎回分ける運用なら、差し替え前の `SaveAsText` 退避は必須ではありません。

理由:

- 差し替え前の状態は元のコピーに残っている。
- 失敗したコピーは破棄すればよい。
- DBコピー単位で戻る方が単純。

ただし、次の場合は `SaveAsText` 退避が有効です。

- 差分確認をしたい。
- フォーム単体で戻したい。
- 証跡を残したい。
- 作業コピーを再利用せざるを得ない。

## LoadFromText失敗時の考え方

`LoadFromText` 自体が常に不安定というより、失敗条件が分かりにくいAPIです。

よくある原因:

- 同名オブジェクトが残っている。
- Accessプロセスが残っている。
- `.laccdb` が残っている。
- フォームのコードビハインドの残骸がある。
- 直前の削除失敗で作業コピーが中途半端になっている。
- サンドボックスや権限不足でCOM操作が制限されている。

失敗したら、そのコピーで修復を続けず、新しい作業コピーからやり直します。
